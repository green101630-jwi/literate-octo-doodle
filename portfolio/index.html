OMX Dual-arm Teleoperation & Demonstration Dataset Pipeline

실제 ROBOTIS OMX 양팔 teleoperation을 Isaac Sim의 dual-arm scene과 연결하고, 사람이 조작한 trajectory를 Isaac Lab recorder 기반 HDF5 demonstration dataset으로 저장하기 위해 구축한 real-to-sim 로봇 학습 파이프라인이다.

1. 프로젝트 개요

이 문서는 실제 OMX 양팔 시스템을 구동하는 단계부터 ROS2 topic 확인, Docker container와 host PC 사이의 통신, Isaac Sim scene 연결, teleoperation demonstration 수집까지의 전체 과정을 정리한 기술 문서이다. 단순 실행 명령어 나열이 아니라, 어떤 데이터가 어디서 생성되고 어떤 topic을 통해 simulation으로 전달되는지까지 함께 정리했다.

1.1 구현 범위

Real-to-Sim Teleoperation

실제 OMX leader arm의 joint state를 ROS2 topic으로 받아 Isaac Sim의 dual OMX arm에 실시간 반영하는 흐름을 구성했다.

Dual-arm Republisher

두 개의 leader arm topic을 각각 Isaac Sim용 topic으로 변환하는 republisher 실행 구조를 정리했다.

Docker 기반 Robot Bring-up

ROBOTIS open_manipulator 환경을 Docker container 안에서 실행하고, 양팔 teleoperation launch를 구동하는 절차를 정리했다.

HDF5 Demonstration Collection

teleoperation으로 만든 demonstration을 Isaac Lab recorder를 통해 HDF5 dataset으로 저장하는 흐름을 구성했다.

1.2 산출물

  • 실제 양팔 teleoperation을 Isaac Sim의 First_env.usd scene에 반영하는 실행 절차
  • republish_joint_states_for_isaac.py, republish_joint_states_for_isaac_2.py 기반 dual-arm topic bridge 실행 흐름
  • collect_omx_live_teleop_hdf5_only.py 기반 HDF5 demonstration 수집 절차
  • DDS, QoS, topic, prim path, container shell 기준의 오류 진단 체크리스트

1.3 결과 요약

실제 leader arm 움직임을 Isaac Sim의 두 OMX arm에 반영하고, 사람이 수행한 teleoperation trajectory를 imitation learning 학습에 사용할 수 있는 HDF5 demonstration dataset으로 저장하는 파이프라인을 구성했다.

2. 시스템 구조

2.1 전체 목표

이 시스템의 목표는 실제 OMX 양팔 조작 데이터를 simulation으로 실시간 전달하고, 같은 흐름에서 HDF5 demonstration dataset을 생성해 imitation learning 또는 reinforcement learning 실험으로 확장할 수 있게 만드는 것이다.

2.2 전체 데이터 흐름

Real OMX leader arms → ROS2 joint_states → Docker/Host DDS 통신 → Republisher → Isaac Sim dual OMX arms → Isaac Lab Recorder → HDF5 dataset

2.3 단계별 구성

1. 실제 OMX사람이 leader arm을 직접 조작
2. ROS2 Containeropen_manipulator launch가 joint state topic 발행
3. Host PCDDS를 통해 container topic 수신
4. RepublisherIsaac Sim용 topic으로 joint state 재발행
5. Isaac Simdual OMX arm이 실제 움직임을 반영
6. Datasetteleoperation demo를 HDF5로 저장
7. LearningBC/RL policy 학습으로 확장

2.4 주요 구성 요소

실제 OMX 로봇팔

사람이 직접 움직이는 leader arm과 실제 작업을 수행하는 follower arm으로 구성된다.

ROS2 Teleoperation Container

ROBOTIS open_manipulator 기반 launch를 실행하고 pair1, pair2 topic을 생성한다.

Host PC

Isaac Sim을 실행하고 container에서 발행되는 ROS2 topic을 수신한다.

Isaac Sim / Isaac Lab

simulation scene을 구성하고, teleoperation 데이터를 HDF5 dataset으로 저장한다.

4. ROS2와 Docker 기본 이해

4.1 ROS2의 역할

ROS2는 실제 로봇, simulation, 데이터 수집 script 사이에서 상태와 command를 주고받기 위한 통신 구조이다. 이 프로젝트에서는 leader arm의 joint state, follower arm command, gripper 상태, simulation 입력 topic 등이 ROS2 topic으로 전달된다.

4.2 Docker를 사용하는 이유

Docker는 ROBOTIS open_manipulator 실행 환경을 host PC와 분리하기 위해 사용한다. 이를 통해 ROS2 package, robot bring-up script, dependency를 안정적으로 유지할 수 있다.

주의: Docker 안에서 보이는 ROS2 topic이 host에서 바로 보이지 않을 수 있다. DDS 설정, network mode, ROS_DOMAIN_ID, QoS가 맞아야 host Isaac Sim에서 topic을 정상적으로 받을 수 있다.

5. Real-to-Sim Teleoperation 실행 절차

아래 절차는 실제 시스템에서 사용한 bring-up 순서를 기준으로 정리한 것이다. 터미널을 여러 개 열어두고 각 터미널에서 역할별 명령어를 실행한다.

환경 분리: republisher는 env_isaaclab을 켜지 않은 host ROS2 환경에서 실행한다. Isaac Lab script와 학습 관련 코드는 env_isaaclab을 켠 상태에서 실행한다.

터미널 1첫 번째 arm republisher 실행env_isaaclab 비활성
터미널 2두 번째 arm republisher 실행env_isaaclab 비활성
터미널 3Docker 권한 확인 및 open_manipulator container 실행env_isaaclab 활성

5.2 터미널 1: 첫 번째 arm republisher 실행

첫 번째 republisher는 /pair1/leader/joint_states를 받아서 Isaac Sim용 topic인 /sim/omx_f/joint_states로 다시 publish한다.

Host 터미널 1, env_isaaclab 비활성 상태 
conda deactivate
cd ~/IsaacLab
python3 republish_joint_states_for_isaac.py
터미널 유지 필요: republisher를 실행한 터미널은 계속 켜 둔다. 이 터미널을 닫거나 다른 명령어를 실행하면 Isaac Sim으로 joint state가 전달되지 않는다.

5.3 터미널 2: 두 번째 arm republisher 실행

두 번째 republisher는 /pair2/leader/joint_states를 받아서 Isaac Sim용 topic인 /sim/omx_f2/joint_states로 다시 publish한다.

Host 터미널 2, env_isaaclab 비활성 상태 
conda deactivate
cd ~/IsaacLab
python3 republish_joint_states_for_isaac_2.py
터미널 유지 필요: 두 republisher 터미널은 계속 실행 중이어야 한다. 터미널을 닫거나 다른 작업에 사용하면 simulation으로 topic이 전달되지 않는다.

5.4 터미널 3: Isaac Lab conda 환경 활성화

Docker 명령 자체는 conda 환경 없이도 실행할 수 있다. 다만 이후 Isaac Lab script와 학습 코드 실행까지 이어지므로, 이 문서에서는 Docker 실행용 터미널에서 env_isaaclab을 활성화한 상태로 진행한다.

Host 터미널 3 
conda activate env_isaaclab

5.5 터미널 3: Docker 권한 설정 및 확인

Docker를 일반 사용자 권한으로 실행하기 위한 설정이다. 이미 docker ps가 정상 실행되는 PC라면 sudo usermod와 newgrp docker는 매번 실행할 필요가 없다.

5.6 터미널 3: open_manipulator container 시작

실제 OMX 양팔 teleoperation은 open_manipulator container 안에서 실행한다.

5.7 별도 진행: physical_ai_tools container 시작

별도 진행 단계: 이 단계는 open_manipulator container 내부에서 실행하지 않는다. 새 host 터미널을 따로 열어서 진행한다.
Host 터미널 4

5.8 터미널 3: 실제 로봇 전선 연결 후 양팔 teleoperation launch 실행

실행 전 전선 연결: launch를 실행하기 전에 실제 follower와 leader 로봇 전선을 컴퓨터에 연결한다. 전선에 표시한 번호 기준으로 2번을 먼저 연결하고, 그 다음 1번을 연결한다.
open_manipulator container 내부

정상 결과: launch가 정상 실행되면 /pair1, /pair2 namespace의 ROS2 topic들이 생성된다.

  1. Stage에 /World/omx_f, /World/omx_f2가 있는지 확인한다.
  2. Action Graph의 topic이 /sim/omx_f/joint_states, /sim/omx_f2/joint_states로 연결되어 있는지 확인한다.
  3. Isaac Sim 왼쪽에 있는 Play 버튼을 누른다.

6. Host PC와 container 사이 ROS2 통신 설정

6.1 설정이 필요한 이유

Isaac Sim은 host PC에서 실행되지만, teleoperation launch는 Docker container 안에서 실행된다. 따라서 Isaac Sim이 container 안의 ROS2 topic을 읽으려면 host와 container 사이의 DDS 통신이 정상이어야 한다.

6.2 host PC 환경변수 설정

Host 터미널 
export ROS_DOMAIN_ID=30
export RMW_IMPLEMENTATION=rmw_fastrtps_cpp
export FASTRTPS_DEFAULT_PROFILES_FILE=~/fastdds.xml
  • ROS_DOMAIN_ID는 container와 host가 같아야 한다.
  • FASTRTPS_DEFAULT_PROFILES_FILE은 실제 파일 경로와 일치해야 한다.
  • fastdds.xml은 DDS가 사용할 네트워크 인터페이스를 제한하기 위해 사용한다.

6.3 ROS2 daemon 재시작

Host 터미널 
ros2 daemon stop
ros2 daemon start

필요하면 다음 폴더를 삭제한 뒤 다시 시도한다.

Host 터미널 
rm -rf ~/.ros

7. QoS와 Republisher

7.1 QoS가 중요한 이유

ROS2에서는 topic 이름과 message type이 같아도 QoS 설정이 맞지 않으면 subscriber가 데이터를 받지 못할 수 있다. 특히 leader joint state topic이 RELIABLE, TRANSIENT_LOCAL로 발행되는 경우가 있어 Isaac Sim subscriber와 직접 맞지 않을 수 있다.

7.2 QoS 확인 명령어

Host 또는 Docker container 내부 
ros2 topic info /pair2/leader/joint_states -v

7.3 Republisher의 역할

  1. 원본 leader joint state topic을 구독한다.
  2. Isaac Sim이 원하는 joint name 순서로 재정렬한다.
  3. gripper joint 값을 simulation 환경에 맞게 변환한다.
  4. Isaac Sim용 topic으로 다시 publish한다.

7.4 topic 매핑

입력 topic republisher 출력 topic Isaac Sim prim
/pair1/leader/joint_states republish_joint_states_for_isaac.py /sim/omx_f/joint_states /World/omx_f
/pair2/leader/joint_states republish_joint_states_for_isaac_2.py /sim/omx_f2/joint_states /World/omx_f2

9. HDF5 Demonstration 수집

9.1 수집 방식

이 단계는 사람이 실제 OMX leader arm을 직접 움직여 demonstration을 만드는 과정이다. scripted motion이나 motion planning으로 자동 경로를 만드는 것이 아니라, teleoperation으로 사람이 직접 수행한 demo를 저장한다.

9.2 사용하는 파일

collect_omx_live_teleop_hdf5_only.py

이 script는 live teleoperation으로 들어오는 양팔 joint state를 읽고, Isaac Lab 환경 안에서 action으로 변환한 뒤, 성공한 episode만 HDF5 demo로 저장한다.

9.3 기본 저장 경로와 파일명

~/IsaacLab/omx_related_files/datasets/omx_live_teleop_direct.hdf5

저장 폴더는 ~/IsaacLab/omx_related_files/datasets/이고, 기본 파일 이름은 omx_live_teleop_direct.hdf5이다. 다른 이름으로 저장하려면 실행 시 --dataset_file argument를 바꾼다.

9.4 기본 task와 입력 topic

항목 기본값
Isaac Lab taskIsaac-Lift-TwoCube-OMX-Dual2-v0
왼쪽 real arm joint state topic/pair1/joint_states
오른쪽 real arm joint state topic/pair2/joint_states
수집할 demo 개수20
stream 주기30 Hz

stream 주기의 의미

stream_hz는 실제 teleoperation에서 들어오는 양팔 joint state를 수집 script가 몇 Hz로 읽어올지 정하는 값이다. 기본값 30 Hz는 1초에 약 30번 real robot의 joint position을 읽어서 Isaac Lab 환경에 action으로 적용한다는 뜻이다. 즉, 대략 33 ms마다 한 번씩 teleop sample을 처리한다.

처음 수집할 때는 기본값 30 Hz를 유지하고, 움직임이 끊기거나 시스템 부하가 큰 경우에만 조정하는 것이 안전하다.

9.7 수집 중 사용하는 키

키 기능 설명
sstart새 episode recording을 시작한다.
eend / save현재 episode를 끝내고 HDF5 demo로 저장한다. 최소 step 수를 만족해야 저장된다.
ddiscard현재 episode를 저장하지 않고 버린다.
pprint latest현재 들어오는 최신 teleop sample을 터미널에 출력한다.
qquit수집 script를 종료한다.

9.8 실제 수집 순서

  1. 양팔 teleoperation이 정상적으로 움직이는지 확인한다.
  2. 데모 수집 script를 실행한다.
  3. 터미널에 Commands: s=start, e=end/save, d=discard, p=print latest, q=quit가 출력되는지 확인한다.
  4. s를 입력하고 Enter를 눌러 recording을 시작한다.
  5. 사람이 직접 leader arm을 움직여 원하는 작업을 수행한다.
  6. 작업이 끝나면 e를 입력하고 Enter를 눌러 episode를 저장한다.
  7. 실패한 demo라면 d를 입력해서 저장하지 않고 버린다.
  8. 목표 demo 개수만큼 반복한다.
  9. 수집을 끝내려면 q를 입력한다.

9.9 저장 조건

기본 설정에서는 episode가 너무 짧으면 저장되지 않는다. script 안의 --min_episode_steps 기본값은 20이며, recording 중 적용된 step 수가 이 값보다 작으면 e를 눌러도 저장되지 않는다.

10. Imitation Learning 확장 계획

현재 파이프라인의 핵심 산출물은 teleoperation 기반 HDF5 demonstration dataset이다. 이 dataset은 이후 observation-action pair 기반 behavior cloning 학습의 입력으로 사용할 수 있다.

10.1 Behavior Cloning으로 연결되는 방식

HDF5 demonstration → observation/action pair 추출 → policy network 학습 → Isaac Sim 평가 → 안전 조건 검증 후 real robot 적용 검토

10.2 학습 입력과 출력

구분 내용
입력HDF5 demonstration 안의 observation
정답teleoperation으로 기록된 expert action
학습 목표현재 observation에서 expert action과 유사한 action을 출력하는 policy 학습
평가 환경Isaac Sim / Isaac Lab task

10.3 BC의 한계

Behavior Cloning은 expert data 분포 안에서는 잘 동작할 수 있지만, policy가 한 번 실수해서 expert data에 없는 상태로 들어가면 복구 행동을 알기 어렵다. 이 문제를 distribution shift라고 한다.

10.4 RL 또는 fine-tuning 확장

teleoperation dataset으로 BC pretraining을 수행한 뒤, Isaac Sim 환경에서 reward 기반 fine-tuning을 적용할 수 있다. 다만 RL은 reward 설계와 학습 안정성 문제가 있으므로, real robot 적용 전에는 simulation 평가와 안전 조건 검증이 필요하다.

11. 오류 진단

11.1 host에서 container topic이 안 보임

  • ROS_DOMAIN_ID가 container와 host에서 같은지 확인한다.
  • Docker가 host network를 사용하는지 확인한다.
  • RMW_IMPLEMENTATION이 맞는지 확인한다.
  • FASTRTPS_DEFAULT_PROFILES_FILE 경로가 맞는지 확인한다.
  • ros2 daemon stop, ros2 daemon start를 실행한다.

11.2 topic은 보이는데 echo가 안 됨

  • QoS reliability와 durability가 subscriber와 호환되는지 확인한다.
  • durability가 TRANSIENT_LOCAL인지 확인한다.
  • message type이 sensor_msgs/msg/JointState인지 확인한다.

11.3 Isaac Sim에서 로봇이 움직이지 않음

  • Isaac Sim 왼쪽 Play 버튼을 눌렀는지 확인한다.
  • ROS2 Context의 domain id가 맞는지 확인한다.
  • Subscribe JointState topic 이름이 /sim/omx_f/joint_states, /sim/omx_f2/joint_states인지 확인한다.
  • Articulation Controller target prim이 /World/omx_f, /World/omx_f2인지 확인한다.
  • USD 안의 robot prim이 articulation root를 가지고 있는지 확인한다.
  • joint name 순서가 USD와 일치하는지 확인한다.

11.4 gripper 동작이 이상함

  • gripper joint 이름이 맞는지 확인한다.
  • mimic joint 설정과 직접 제어 joint 설정이 충돌하지 않는지 확인한다.
  • gripper scale, sign, offset이 실제 방향과 맞는지 확인한다.
  • gripper joint limit이 simulation 설정과 맞는지 확인한다.

11.5 Isaac Sim이 느리거나 멈춤

  • GPU driver와 VRAM 사용량을 확인한다.
  • 불필요한 asset이 과도하게 load되어 있는지 확인한다.
  • physics timestep과 rendering quality를 낮춰본다.
  • 필요하면 headless 실행을 검토한다.

13. 결과 및 후속 개선 방향

13.1 현재 구현 결과

  • ROBOTIS OMX 양팔 teleoperation을 Docker 기반 open_manipulator 환경에서 실행하는 절차를 정리했다.
  • 두 개의 leader arm joint state를 각각 Isaac Sim용 topic으로 변환하는 republisher 구조를 구성했다.
  • Isaac Sim의 First_env scene에서 실제 teleoperation 움직임을 dual OMX arm에 반영하는 흐름을 정리했다.
  • teleoperation으로 만든 demonstration을 HDF5 dataset으로 저장하는 절차를 구성했다.
  • DDS, QoS, topic, prim path 문제를 단계적으로 진단할 수 있는 checklist를 정리했다.

13.2 후속 개선 방향

항목 개선 방향
Dataset 검증 HDF5 내부 demo 개수, action range, episode length, success flag를 자동 확인하는 검증 script 추가
Behavior Cloning 학습 수집된 HDF5 dataset을 사용한 BC training script와 checkpoint 저장 경로 정리
Policy 평가 Isaac Sim에서 성공률, episode return, failure case를 기록하는 evaluation 절차 추가
실제 로봇 적용 안전성 joint limit, low-speed command, emergency stop, workspace 제한 조건을 포함한 real robot 실행 절차 추가
시각 자료 실제 OMX 사진, Isaac Sim scene, ROS2 topic echo, HDF5 저장 결과 화면을 포트폴리오에 추가
이 문서는 실제 실행 절차와 시스템 구조를 함께 정리한 포트폴리오형 기술 문서이다. 후속 단계에서는 수집된 HDF5 dataset을 이용한 policy 학습 결과와 정량 평가를 추가하면 연구 포트폴리오로서 완성도가 더 높아진다.
