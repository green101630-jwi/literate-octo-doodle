3. ROS2와 Docker 기본 이해

3.1 ROS2가 하는 일

ROS2는 로봇 시스템에서 여러 프로그램이 데이터를 주고받게 해주는 통신 구조이다. teleoperation 시스템에서는 leader arm의 현재 관절 각도, follower arm으로 보낼 목표 관절 각도, gripper 상태, 카메라 이미지, 로봇 상태 정보 등이 ROS2 topic으로 흐른다.

3.2 Docker를 쓰는 이유

Docker는 로봇 구동 환경을 하나의 독립된 실행 환경으로 고정하기 위해 사용한다. ROBOTIS에서 제공한 환경을 그대로 사용할 수 있고, host PC의 ROS2 버전과 container 내부 ROS2 버전이 달라도 관리할 수 있다.

주의: Docker 안에서 보이는 ROS2 topic이 host에서 바로 보이지 않을 수 있다. DDS 설정, network mode, ROS_DOMAIN_ID, QoS를 확인해야 한다.

4. 실제 Teleoperation 시스템 실행 흐름

아래 순서는 현재 실험실 시스템에서 실제로 수행한 절차를 기준으로 정리한 것이다. 처음 실행하는 학생은 터미널을 여러 개 열어두고, 각 터미널에서 아래 명령어를 순서대로 실행하면 된다.

중요: republisher는 env_isaaclab을 켜지 않은 host ROS2 환경에서 실행한다. 반대로 Isaac Lab 관련 script나 학습 코드 실행은 env_isaaclab을 켠 상태에서 진행한다.

4.1 전체 실행에 필요한 터미널 구성

터미널 역할 conda 상태
터미널 1 첫 번째 arm republisher 실행 env_isaaclab 비활성
터미널 2 두 번째 arm republisher 실행 env_isaaclab 비활성
터미널 3 Docker 권한 확인 및 open_manipulator container 실행 env_isaaclab 활성
터미널 4 별도 진행: physical_ai_tools container 실행 env_isaaclab 활성 또는 비활성 모두 가능
터미널 5 Isaac Sim 실행 필요 시 env_isaaclab 활성

4.2 터미널 1: 첫 번째 arm republisher 실행

첫 번째 republisher는 /pair1/leader/joint_states를 받아서 Isaac Sim용 topic인 /sim/omx_f/joint_states로 다시 publish한다.

Host 터미널 1, env_isaaclab 비활성 상태 
conda deactivate
cd ~/IsaacLab
python3 republish_joint_states_for_isaac.py
conda deactivate를 실행했을 때 이미 conda 환경이 꺼져 있다면 큰 문제는 없다. 핵심은 이 republisher를 env_isaaclab 안에서 실행하지 않는 것이다.
건드리지 말 것: republisher를 실행한 터미널은 계속 켜 둔다. 이 터미널에서 다른 명령어를 치거나 터미널을 닫으면 Isaac Sim으로 joint state가 전달되지 않는다.

4.3 터미널 2: 두 번째 arm republisher 실행

두 번째 republisher는 /pair2/leader/joint_states를 받아서 Isaac Sim용 topic인 /sim/omx_f2/joint_states로 다시 publish한다.

Host 터미널 2, env_isaaclab 비활성 상태 
conda deactivate
cd ~/IsaacLab
python3 republish_joint_states_for_isaac_2.py
건드리지 말 것: 두 republisher 터미널은 계속 실행 중이어야 한다. 실행 후 터미널을 닫거나 다른 작업에 사용하면 Isaac Sim으로 joint state가 전달되지 않는다.

4.4 터미널 3: Isaac Lab conda 환경 활성화

Docker 실행 자체는 conda 환경이 없어도 가능하지만, 이후 Isaac Lab script와 학습 코드 실행까지 이어지므로 이 안내서에서는 Docker 실행용 터미널에서 env_isaaclab을 활성화한 상태로 진행한다.

Host 터미널 3 
conda activate env_isaaclab

4.5 터미널 3: Docker 권한 설정 및 확인

Docker를 일반 사용자 권한으로 실행하기 위한 설정이다. 이미 docker ps가 정상 실행되는 PC라면 sudo usermod와 newgrp docker는 매번 할 필요 없다.

Host 터미널 3 
sudo usermod -aG docker $USER
newgrp docker
docker ps

sudo 명령에서 비밀번호를 물어보면 Ubuntu 로그인 비밀번호를 입력한다. docker ps가 오류 없이 container 목록 표를 출력하면 Docker 권한은 정상이다.

4.6 터미널 3: open_manipulator container 시작

실제 OMX 양팔 teleoperation은 open_manipulator container 안에서 실행한다.

Host 터미널 3 
cd ~/open_manipulator/docker
./container.sh start
./container.sh enter
정상 결과: ./container.sh enter 이후에는 프롬프트가 host가 아니라 container 내부 shell로 바뀐다. 화면에 root@bio:~# 또는 root@bio:/ros2_ws#처럼 표시되면 container 내부에 들어온 상태이다.

4.7 별도 진행: physical_ai_tools container 시작

별도 진행 단계: 이 단계는 open_manipulator container 안에서 하는 작업이 아니다. 새 host 터미널을 따로 열어서 진행한다.

physical AI 관련 관리 도구 container가 필요한 경우 별도 host 터미널에서 실행한다. 이 명령은 open_manipulator container 안이 아니라 host에서 실행한다.

실행 전 전선 연결: launch를 실행하기 전에 실제 follower와 leader 로봇 전선을 컴퓨터에 연결한다. 전선에 적힌 번호 기준으로 2번을 먼저 연결하고, 그 다음 1번을 연결한다.

잘 안 될 때 확인하는 단계: 이 단계는 매번 필수로 실행하는 단계가 아니다. 실제 로봇을 움직였는데 Isaac Sim이 따라오지 않거나 topic이 의심될 때 확인한다.

5. Host PC와 container 사이 ROS2 통신 설정

5.1 왜 설정이 필요한가

Isaac Sim은 host PC에서 실행된다. 하지만 teleoperation launch는 Docker container 안에서 실행된다. 따라서 Isaac Sim이 container 안의 ROS2 topic을 읽으려면 host와 container 사이의 DDS 통신이 정상이어야 한다.

5.2 host PC 환경변수 설정

Host 터미널 
export ROS_DOMAIN_ID=30
export RMW_IMPLEMENTATION=rmw_fastrtps_cpp
export FASTRTPS_DEFAULT_PROFILES_FILE=/home/bio/fastdds.xml
  • ROS_DOMAIN_ID는 container와 host가 같아야 한다.
  • FASTRTPS_DEFAULT_PROFILES_FILE은 실제 파일 경로와 일치해야 한다.
  • fastdds.xml은 DDS가 사용할 네트워크 인터페이스를 제한하기 위해 사용한다.

5.3 ROS2 daemon 재시작

Host 터미널 
ros2 daemon stop
ros2 daemon start

필요하면 다음 폴더를 삭제한 뒤 다시 시도한다.

Host 터미널 
rm -rf ~/.ros

5.4 host에서 topic 확인

Host 터미널 
ros2 topic list
정상 결과: Docker container 안에서 보이던 /pair1/leader/joint_states, /pair2/leader/joint_states가 host에서도 보여야 한다.

6. QoS 확인과 Republisher 필요성

6.1 QoS가 중요한 이유

ROS2에서는 topic 이름과 message type이 같아도 QoS 설정이 맞지 않으면 subscriber가 데이터를 받지 못할 수 있다. 특히 leader joint state topic이 RELIABLE, TRANSIENT_LOCAL로 발행되는 경우가 있다.

6.2 QoS 확인 명령어

Host 또는 Docker container 내부 
ros2 topic info /pair2/leader/joint_states -v

확인할 항목

  • Type
  • Reliability
  • Durability
  • Depth
  • Publisher count
  • Subscription count

6.3 Republisher를 사용하는 이유

원본 topic의 QoS나 joint name 순서가 Isaac Sim과 맞지 않을 수 있다. 이 경우 custom republisher node를 사용해 Isaac Sim이 읽기 쉬운 형태로 다시 publish한다.

Republisher가 하는 일

  1. 원본 leader joint state topic을 구독한다.
  2. Isaac Sim이 원하는 joint name 순서로 재정렬한다.
  3. gripper joint 값을 변환한다.
  4. Isaac Sim용 topic으로 다시 publish한다.

예시 output topic

/sim/omx_f/joint_states
/sim/omx_f2/joint_states

8. Teleoperation 데모 수집과 HDF5 저장

8.1 이 단계에서 하는 일

이 단계는 사람이 실제 OMX leader arm을 직접 움직여서 demonstration을 만드는 과정이다. 즉, script나 motion planning으로 자동 경로를 만드는 것이 아니라, teleoperation으로 사람이 직접 만든 demo를 저장한다.

수집된 demo는 Isaac Lab recorder를 통해 HDF5 파일로 저장된다. 이후 behavior cloning 같은 imitation learning에서 이 HDF5 파일을 expert dataset으로 사용할 수 있다.

8.2 사용하는 파일

데모 수집에는 다음 Python 파일을 사용한다.

collect_omx_live_teleop_hdf5_only.py

이 파일은 live teleoperation으로 들어오는 양팔 joint state를 읽고, Isaac Lab 환경 안에서 action으로 변환한 뒤, 성공한 episode만 HDF5 demo로 저장하는 script이다.

8.3 기본 저장 경로

파일 안의 기본 HDF5 저장 경로는 다음과 같다.

/home/bio/IsaacLab/omx_related_files/datasets/omx_live_teleop_direct.hdf5

다른 이름으로 저장하고 싶으면 실행할 때 --dataset_file argument를 바꾸면 된다.

stream 주기 30 Hz

stream 주기의 의미

stream_hz는 실제 teleoperation에서 들어오는 양팔 joint state를 수집 script가 몇 Hz로 읽어올지 정하는 값이다. 기본값 30 Hz는 1초에 약 30번 real robot의 joint position을 읽어서 Isaac Lab 환경에 action으로 적용한다는 뜻이다.

예를 들어 30 Hz이면 시간 간격은 대략 1 / 30 = 0.033초, 즉 약 33 ms마다 한 번씩 teleop sample을 처리한다. 이 값이 너무 낮으면 움직임이 끊겨 보일 수 있고, 너무 높으면 ROS2 통신, Python 처리, Isaac Sim step 처리 부담이 커질 수 있다.

현재 기본 설정: 이 안내서에서는 30 Hz를 기본값으로 사용한다. 처음 수집할 때는 기본값을 유지하고, 움직임이 끊기거나 시스템 부하가 큰 경우에만 조정하는 것이 안전하다.

stream 주기 변경 예시

Host 터미널, env_isaaclab 활성 상태 
cd ~/IsaacLab
./isaaclab.sh -p omx_related_files/scripts/custom/collect_omx_live_teleop_hdf5_only.py \
  --stream_hz 20.0

위 예시는 teleop sample 처리 주기를 20 Hz로 낮춘 것이다. 반대로 더 부드러운 입력을 원하면 --stream_hz 40.0처럼 높일 수 있지만, 실제로는 Isaac Sim과 ROS2 처리 상태를 보면서 조정해야 한다.

주의: teleop 확인용 republisher는 /pair1/leader/joint_states, /pair2/leader/joint_states를 사용했지만, 이 HDF5 수집 script의 기본 입력 topic은 /pair1/joint_states, /pair2/joint_states이다. 실제 topic 이름이 다르면 실행할 때 --left_joint_states_topic, --right_joint_states_topic으로 바꿔야 한다.

8.8 수집 중 사용하는 단축키

키 기능 설명
s start 새 episode recording을 시작한다.
e end / save 현재 episode를 끝내고 HDF5 demo로 저장한다. 최소 step 수를 만족해야 저장된다.
d discard 현재 episode를 저장하지 않고 버린다.

8.9 실제 수집 순서

  1. 양팔 teleoperation이 정상적으로 움직이는지 확인한다.
  2. 데모 수집 script를 실행한다.
  3. 터미널에 Commands: s=start, e=end/save, d=discard, p=print latest, q=quit가 출력되는지 확인한다.
  4. s를 입력하고 Enter를 눌러 recording을 시작한다.
  5. 사람이 직접 leader arm을 움직여 원하는 작업을 수행한다.
  6. 작업이 끝나면 e를 입력하고 Enter를 눌러 episode를 저장한다.
  7. 실패한 demo라면 d를 입력해서 저장하지 않고 버린다.
  8. 목표 demo 개수만큼 반복한다.
  9. 수집을 끝내려면 q를 입력한다.

8.10 저장 조건

기본 설정에서는 episode가 너무 짧으면 저장되지 않는다. script 안의 --min_episode_steps 기본값은 20이며, recording 중 적용된 step 수가 이 값보다 작으면 e를 눌러도 저장되지 않는다.

9. Behavior Cloning 학습 흐름

9.1 Behavior Cloning이란

Behavior Cloning은 expert가 특정 상태에서 한 행동을 그대로 supervised learning으로 학습하는 imitation learning 방법이다. 

입력: observation
정답: expert action
목표: observation을 넣었을 때 expert action과 비슷한 action을 출력하는 policy 학습

9.2 학습 순서

  1. teleoperation으로 demo를 수집한다.
  2. demo를 HDF5 또는 dataset 형식으로 저장한다.
  3. dataset을 train / validation으로 분리한다.
  4. policy network를 정의한다.
  5. observation을 입력으로 넣고 action을 예측한다.
  6. expert action과 predicted action 사이 loss를 계산한다.
  7. loss가 줄어들도록 policy를 학습한다.
  8. 학습된 policy를 Isaac Sim에서 평가한다.
  9. 가능하면 real robot에서 제한적으로 검증한다.

9.3 BC의 한계

BC는 expert data 분포 안에서는 잘 동작할 수 있다. 하지만 로봇이 학습 중 실수로 expert data에서 벗어난 상태에 들어가면, 그 상태에서 무엇을 해야 할지 모를 수 있다. 이 문제를 distribution shift라고 한다.

10. Reinforcement Learning으로 확장

10.1 RL이 필요한 경우

RL은 policy가 직접 환경과 상호작용하면서 reward를 최대화하도록 학습하는 방식이다. teleoperation demo만으로 부족하거나, 더 안정적인 정책을 만들고 싶을 때 RL로 확장할 수 있다.

10.2 BC와 RL의 차이

구분 Behavior Cloning Reinforcement Learning
학습 기준 expert action을 정답으로 사용 reward를 최대화
데이터 offline demo dataset 환경과의 interaction 필요
장점 구현과 학습이 비교적 단순함 실패 상황을 직접 경험하며 개선 가능
단점 data 분포 밖 상황에 약함 reward 설계와 학습 안정성이 어려움

10.3 가능한 학습 흐름

teleop demo 수집 → BC pretraining → Isaac Sim에서 policy 평가 → RL fine-tuning → real robot 검증

11. 자주 발생하는 오류와 해결 방향

11.1 host에서 container topic이 안 보임

  • ROS_DOMAIN_ID가 같은가?
  • Docker가 host network를 쓰는가?
  • RMW_IMPLEMENTATION이 맞는가?
  • FastDDS XML 경로가 맞는가?
  • ros2 daemon을 재시작했는가?

11.2 topic은 보이는데 echo가 안 됨

  • QoS가 맞는가?
  • durability가 TRANSIENT_LOCAL인지 확인했는가?
  • message type이 sensor_msgs/msg/JointState인지 확인했는가?

11.3 Isaac Sim에서 로봇이 안 움직임

  • Play 버튼을 눌렀는가?
  • ROS2 Context domain이 30인가?
  • Subscribe JointState topic 이름이 맞는가?
  • Articulation Controller target prim이 맞는가?
  • robot prim이 articulation root를 가지고 있는가?
  • joint name 순서가 USD와 일치하는가?

11.4 gripper가 이상하게 움직임

  • gripper joint 이름이 맞는가?
  • mimic joint 설정이 충돌하지 않는가?
  • gripper scale과 sign이 맞는가?
  • gripper joint limit이 맞는가?

11.5 Isaac Sim이 멈추거나 느림

  • GPU driver 상태
  • VRAM 사용량
  • headless 여부
  • loaded asset 개수
  • physics timestep
  • rendering quality

  • /World/omx_f, /World/omx_f2 prim path가 맞는가?
  • Isaac Sim 왼쪽의 Play 버튼을 눌렀는가?
  • 실제 양팔 teleop 움직임이 sim의 두 arm에 적용되는가?
